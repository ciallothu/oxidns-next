This page describes the OxiDNS Next installation paths and the shortest path from download to first successful start.

<Admonition type="tip" title="Which install path should I choose?">

* **Linux servers**: the one-command installer, release archives, or `.deb` packages
* **OpenWrt routers**: a musl archive matching the device architecture; the current [portable deployment](openwrt.mdx) has no dedicated LuCI app
* **Containerized environments**: the Docker image
* **Local development or debugging**: build from source with Cargo
* **macOS / Windows**: the one-command installer or release archives

</Admonition>

## 1. One-Command Install

The installer detects the current platform, downloads the matching OxiDNS Next GitHub Release archive, and extracts `oxidns-next` / `oxidns-next.exe`, `config.yaml`, and `webui/`. Linux, macOS, and elevated Windows installs start a system service by default; OpenWrt uses portable mode only.

<Tabs groupId="os">
  <TabItem value="unix" label="Linux / macOS" default>

```bash
curl -fsSL https://raw.githubusercontent.com/ciallothu/oxidns-next/main/scripts/install.sh | sudo sh
```

  </TabItem>
  <TabItem value="windows" label="Windows">

```powershell
irm https://raw.githubusercontent.com/ciallothu/oxidns-next/main/scripts/install.ps1 | iex
```

  </TabItem>
  <TabItem value="openwrt" label="OpenWrt">

```sh
curl -fsSL https://raw.githubusercontent.com/ciallothu/oxidns-next/main/scripts/install.sh | sh
```

If `curl` is not installed:

```sh
wget -O- https://raw.githubusercontent.com/ciallothu/oxidns-next/main/scripts/install.sh | sh
```

  </TabItem>
</Tabs>

Default install locations:

| Platform | Default install directory | PATH behavior |
| --- | --- | --- |
| Linux / macOS service install with `sudo` | `/opt/oxidns-next` | Creates `/usr/local/bin/oxidns-next`, then installs and starts the system service |
| Windows service install from elevated PowerShell | `%ProgramFiles%\OxiDNS Next` | Adds the install directory to the Machine PATH, then installs and starts the service |
| Linux / macOS portable install | `~/.oxidns-next` | Set `OXIDNS_NEXT_INSTALL_SERVICE=0`; creates `~/.local/bin/oxidns-next` |
| Windows portable install | `%LOCALAPPDATA%\OxiDNS Next` | Set `OXIDNS_NEXT_INSTALL_SERVICE=0`; adds the install directory to the user PATH |
| OpenWrt portable install | `/opt/oxidns-next` | Creates `/usr/local/bin/oxidns-next` without registering an init service |

In Linux / macOS / Windows archive install mode, running the installer again updates the binary and `webui/`. If `config.yaml` already exists in the install directory, the installer keeps it and writes the release default config as `config.yaml.example`.

On OpenWrt, running the installer again updates the binary and WebUI while preserving the existing `config.yaml`. Configure service supervision separately.

### Install A Specific Version

<Tabs groupId="os">
  <TabItem value="unix" label="Linux / macOS" default>

```bash
curl -fsSL https://raw.githubusercontent.com/ciallothu/oxidns-next/main/scripts/install.sh | sudo env OXIDNS_NEXT_VERSION=v0.1.0 sh
```

  </TabItem>
  <TabItem value="windows" label="Windows">

```powershell
$env:OXIDNS_NEXT_VERSION = "v0.1.0"; irm https://raw.githubusercontent.com/ciallothu/oxidns-next/main/scripts/install.ps1 | iex
```

  </TabItem>
</Tabs>

Common environment variables:

| Variable | Purpose |
| --- | --- |
| `OXIDNS_NEXT_VERSION` | Release tag, defaults to `latest` |
| `OXIDNS_NEXT_INSTALL_DIR` | Override the install directory |
| `OXIDNS_NEXT_BIN_DIR` | Override the command-link directory on Linux / macOS |
| `OXIDNS_NEXT_TARGET` | Override the release target, for example `x86_64-unknown-linux-musl` |
| `OXIDNS_NEXT_BUNDLE` | Select the release build bundle, default `full`; Linux x86_64 / ARM64 musl can use `minimal` or `standard` |
| `OXIDNS_NEXT_NO_PATH=1` | Do not create a command link or update PATH |
| `OXIDNS_NEXT_INSTALL_SERVICE=0` | Skip system service installation and use portable install mode |
| `OXIDNS_NEXT_START_SERVICE=0` | Install the service but do not start it immediately |

<Admonition type="note" title="Port permissions">

The default config includes a `:53` listener. On Linux / macOS, portable installs run by regular users usually cannot bind port 53; for local evaluation, remove `udp_server_53` first or use the default system service install.

</Admonition>

<Admonition type="warning" title="Purge removes configuration">

Setting `OXIDNS_NEXT_PURGE=1` deletes the complete install directory, including its configuration. Back up `config.yaml` before running it.

</Admonition>

Common uninstall variables:

| Variable | Purpose |
| --- | --- |
| `OXIDNS_NEXT_INSTALL_DIR` | Override the uninstall target; keep it aligned with the install command |
| `OXIDNS_NEXT_BIN_DIR` | Override the command-link directory on Linux / macOS |
| `OXIDNS_NEXT_NO_PATH=1` | Do not remove the command link or user PATH entry |
| `OXIDNS_NEXT_UNINSTALL_SERVICE=1` | Force operating-system service uninstall; by default this is attempted automatically as root / administrator |
| `OXIDNS_NEXT_PURGE=1` | Remove the complete install directory, including its configuration |

## 2. Build From Source

This path is suitable for debugging, code changes, or cases where packaged builds are not used.

Requirements:

- A stable Rust toolchain
- A working Cargo build environment

Build and run:

```bash
git clone https://github.com/ciallothu/oxidns-next.git
cd oxidns-next

cargo build --release
./target/release/oxidns-next check -c config.yaml
./target/release/oxidns-next start -c config.yaml
```

Run with debug logging:

```bash
cargo run -- check -c config.yaml
cargo run -- start -c config.yaml -l debug
```

## 3. Install From GitHub Release Archives

The release workflow generates standalone binaries for multiple platforms. Each `full` archive includes:

- `oxidns-next` or `oxidns-next.exe`
- `config.yaml`
- `LICENSE`
- `webui/`

Linux x86_64 / ARM64 musl also get `minimal` and `standard` slim archives. `minimal` only includes `oxidns-next`, `config.yaml`, and `LICENSE`; `standard` also includes WebUI static files, query_recorder, and the upgrade capability. See [Custom Build](./custom-build#preset-capability-matrix) for the full capability differences between preset bundles.

Release page:

- [https://github.com/ciallothu/oxidns-next/releases](https://github.com/ciallothu/oxidns-next/releases)

### Supported Archive Targets

Non-Windows targets use `.tar.gz`:

- `oxidns-next-x86_64-unknown-linux-gnu.tar.gz`
- `oxidns-next-x86_64-unknown-linux-musl.tar.gz`
- `oxidns-next-aarch64-unknown-linux-gnu.tar.gz`
- `oxidns-next-aarch64-unknown-linux-musl.tar.gz`
- `oxidns-next-i686-unknown-linux-musl.tar.gz`
- `oxidns-next-arm-unknown-linux-musleabihf.tar.gz`
- `oxidns-next-x86_64-apple-darwin.tar.gz`
- `oxidns-next-aarch64-apple-darwin.tar.gz`
- `oxidns-next-x86_64-unknown-freebsd.tar.gz`

Linux musl slim archives use a bundle prefix:

- `oxidns-next-minimal-x86_64-unknown-linux-musl.tar.gz`
- `oxidns-next-minimal-aarch64-unknown-linux-musl.tar.gz`
- `oxidns-next-standard-x86_64-unknown-linux-musl.tar.gz`
- `oxidns-next-standard-aarch64-unknown-linux-musl.tar.gz`

Windows targets use `.zip`:

- `oxidns-next-x86_64-pc-windows-msvc.zip`
- `oxidns-next-i686-pc-windows-msvc.zip`
- `oxidns-next-aarch64-pc-windows-msvc.zip`

### How To Choose The Right Release Asset

When the correct asset is unclear, use this mapping:

| System / Environment | Recommended release asset | Notes |
| --- | --- | --- |
| Linux x86_64 | `oxidns-next-x86_64-unknown-linux-musl.tar.gz` | Safer default for broad compatibility |
| Linux ARM64 | `oxidns-next-aarch64-unknown-linux-musl.tar.gz` | Safer default for broad compatibility |
| Debian / Ubuntu x86_64 service install | `*_amd64.deb` | Best fit for systemd-based deployment |
| Debian / Ubuntu ARM64 service install | `*_arm64.deb` | Best fit for systemd-based deployment |
| OpenWrt | A Linux musl archive matching the device CPU architecture | The installer uses portable mode; MIPS / MIPS64 currently have no official archive |
| Alpine Linux x86_64 | `oxidns-next-x86_64-unknown-linux-musl.tar.gz` | Prefer musl on Alpine |
| Alpine Linux ARM64 | `oxidns-next-aarch64-unknown-linux-musl.tar.gz` | Static-friendly build |
| Confirmed glibc Linux requiring a dynamic build | `oxidns-next-x86_64-unknown-linux-gnu.tar.gz` / `oxidns-next-aarch64-unknown-linux-gnu.tar.gz` | Only choose this when the target environment is clearly compatible |
| 32-bit ARM Linux | `oxidns-next-arm-unknown-linux-musleabihf.tar.gz` | Fits some Raspberry Pi and older ARM boards |
| macOS Intel | `oxidns-next-x86_64-apple-darwin.tar.gz` | Intel Macs |
| macOS Apple Silicon | `oxidns-next-aarch64-apple-darwin.tar.gz` | M1 / M2 / M3 / M4 Macs |
| Windows x64 | `oxidns-next-x86_64-pc-windows-msvc.zip` | Most PCs |
| Windows 32-bit | `oxidns-next-i686-pc-windows-msvc.zip` | Only for 32-bit Windows |
| Windows ARM64 | `oxidns-next-aarch64-pc-windows-msvc.zip` | ARM-based Windows devices |
| FreeBSD x86_64 | `oxidns-next-x86_64-unknown-freebsd.tar.gz` | FreeBSD hosts |

To verify the platform first, run:

```bash
uname -s
uname -m
```

Common output mapping:

- `Linux` + `x86_64`: default to `x86_64-unknown-linux-musl`; use `x86_64-unknown-linux-gnu` only when a glibc dynamic build is explicitly required
- `Linux` + `aarch64`: default to `aarch64-unknown-linux-musl`; use `aarch64-unknown-linux-gnu` only when a glibc dynamic build is explicitly required
- `Linux` + `armv7l`: use `arm-unknown-linux-musleabihf`
- `Darwin` + `x86_64`: use `x86_64-apple-darwin`
- `Darwin` + `arm64`: use `aarch64-apple-darwin`

On Windows PowerShell, run:

```powershell
(Get-CimInstance Win32_OperatingSystem).OSArchitecture
```

Map `X64`, `Arm64`, and `X86` to the `x86_64`, `aarch64`, and `i686` Windows assets respectively.

### Linux / macOS Example

Replace `TAG` below with the actual release tag, for example `v0.1.0`. The Linux example below uses `x86_64-unknown-linux-musl` as the default choice:

```bash
curl -L -o oxidns-next.tar.gz \
  https://github.com/ciallothu/oxidns-next/releases/download/TAG/oxidns-next-x86_64-unknown-linux-musl.tar.gz

mkdir -p oxidns-next
tar -xzf oxidns-next.tar.gz -C oxidns-next
cd oxidns-next

chmod +x oxidns-next
./oxidns-next check -c config.yaml
./oxidns-next start -c config.yaml
```

When the target machine's glibc compatibility is unclear, or when running Alpine Linux, prefer a `*-linux-musl` archive instead of defaulting to `gnu`.

### Windows Example

Download the matching `.zip`, extract it, then run:

```powershell
.\oxidns-next.exe start -c .\config.yaml
```

## 4. Install From Debian Packages

The release workflow currently builds `.deb` packages for:

- `x86_64-unknown-linux-gnu`
- `aarch64-unknown-linux-gnu`

Install on Debian / Ubuntu:

```bash
sudo dpkg -i oxidns-next_*_amd64.deb
```

Or on ARM64 Debian / Ubuntu:

```bash
sudo dpkg -i oxidns-next_*_arm64.deb
```

Default installed paths:

- Binary: `/usr/bin/oxidns-next`
- Config: `/etc/oxidns-next/config.yaml`
- Runtime-relative path base: `/var/lib/oxidns-next`
- WebUI: `/var/lib/oxidns-next/webui` symlinked to `/usr/share/oxidns-next/webui`

The project also ships systemd packaging metadata, so Debian-family systems are a good fit for service-based deployment. The default service explicitly passes `ExecStart ... -d /var/lib/oxidns-next` as the runtime-relative path base.

After changing the default config, validate it before starting the service:

```bash
oxidns-next check -c /etc/oxidns-next/config.yaml -d /var/lib/oxidns-next
```

Verify service status:

```bash
sudo systemctl status oxidns-next
```

If the service is not running yet:

```bash
sudo systemctl enable --now oxidns-next
```

## 5. Portable OpenWrt Deployment

OpenWrt uses the same release archives as Linux, but the installer skips system-service registration:

```sh
curl -fsSL https://raw.githubusercontent.com/ciallothu/oxidns-next/main/scripts/install.sh | sh
```

If `curl` is not installed:

```sh
wget -O- https://raw.githubusercontent.com/ciallothu/oxidns-next/main/scripts/install.sh | sh
```

After installation, supervise `oxidns-next` yourself through procd or another process manager. Upstream [`luci-app-oxidns`](https://github.com/svenshi/luci-app-oxidns) targets the original OxiDNS distribution and is not this project's installer. See [OpenWrt Deployment](openwrt.mdx) for supported architectures, default paths, and removal instructions.

## 6. Run With Docker

The repository publishes a Docker image at:

- `ghcr.io/ciallothu/oxidns-next`

The Docker workflow builds:

- `linux/amd64`
- `linux/arm64`

Pull and run:

```bash
docker pull ghcr.io/ciallothu/oxidns-next:latest

# Browser requests into a container are not loopback. Use a temporary token
# for the first administrator bootstrap.
export OXIDNS_NEXT_BOOTSTRAP_TOKEN="$(openssl rand -hex 32)"

docker run -d \
  --name oxidns-next \
  --restart unless-stopped \
  -p 53:5335/udp \
  -p 53:5335/tcp \
  -p 9199:9199/tcp \
  -e OXIDNS_NEXT_BOOTSTRAP_TOKEN \
  -v "$(pwd)/config.yaml:/etc/oxidns-next/config.yaml:ro" \
  -v "$(pwd)/data:/etc/oxidns-next/data" \
  ghcr.io/ciallothu/oxidns-next:latest
```

<Admonition type="note" title="Port binding">

On Linux, port 53 is privileged. Regular users usually get the needed permission through the Docker daemon. Portable non-root installs should use `:5353`.

</Admonition>

When the default-branch image is published, the `latest` tag is also available.

The image entrypoint effectively runs:

```bash
oxidns-next start -c /etc/oxidns-next/config.yaml -d /etc/oxidns-next
```

The image metadata declares:

- `53/udp`
- `53/tcp`
- `9199/tcp`

The repository-root `config.yaml` listens for DNS on `:5335` so it can also be run locally without root privileges. The `docker run` command above and Compose example below map host port `53` to container port `5335`. If you change the in-container listener to `:53`, change the mapping back to `53:53` too.

### Docker Compose Example

For Compose-based port mapping and config management, use this `docker-compose.yml`:

```yaml
services:
  oxidns-next:
    image: ghcr.io/ciallothu/oxidns-next:latest
    container_name: oxidns-next
    restart: unless-stopped
    ports:
      - "53:5335/udp"
      - "53:5335/tcp"
      - "9199:9199/tcp"
    environment:
      - OXIDNS_NEXT_BOOTSTRAP_TOKEN
    volumes:
      - ./config.yaml:/etc/oxidns-next/config.yaml:ro
      - ./data:/etc/oxidns-next/data
```

Start it with:

```bash
export OXIDNS_NEXT_BOOTSTRAP_TOKEN="$(openssl rand -hex 32)"
docker compose up -d
```

On the first WebUI visit, enter this token in the setup form and create the administrator. Then immediately remove the temporary token from the container:

```bash
unset OXIDNS_NEXT_BOOTSTRAP_TOKEN
docker compose up -d --force-recreate
```

With a direct `docker run`, remove and recreate the container after setup with the same arguments but without `-e OXIDNS_NEXT_BOOTSTRAP_TOKEN`. Do not place the token in `config.yaml`, the Compose file, or shell history. If OpenSSL is unavailable, use a password manager to generate at least 32 random bytes.

Follow logs with:

```bash
docker compose logs -f oxidns-next
```

The default single-container deployment keeps query history in SQLite. For PostgreSQL, MySQL, or Redis-backed DNS/API caching alongside SQLite, use the Compose examples under [`examples/storage`](https://github.com/ciallothu/oxidns-next/tree/main/examples/storage). They include health checks, environment placeholders, and persistent volumes. Redis is not durable query-history storage.

## 7. Selection Guide

- Fastest evaluation path: use the one-command installer for the latest release, or download a release archive manually
- Long-running Linux service: prefer the Debian package
- OpenWrt deployment: verify the device architecture, use the portable installer, and configure process supervision yourself
- Container platforms: use the GHCR image
- Development or custom builds: compile from source
